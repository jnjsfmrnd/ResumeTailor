# ADR-0001: Round 2 V1 Architecture Decisions

**Status**: Accepted (Planning Only)
**Date**: 2026-04-01
**Author**: GitHub Copilot
**Epic**: #0001

## Context

The user has now tightened V1 scope in round 2:

- Cover letter generation is included as an opt-in mode.
- Hosted deployment is QA only.
- Azure region is `eastus` only.
- Managed PostgreSQL cost and free-tier limits are too restrictive for the current QA budget.
- V1 model access is GitHub Models only.
- Timeout guidance must instruct the user to check their API key and retry.
- The UX design spec is the source of truth for user-facing behavior.

## Decision

We will build a Django-based, PDF-first application with one hosted QA environment in `eastus`, using SQLite for local development, CI, and QA persistence, GitHub Models only for AI access, and an optional cover letter mode.

### Architecture Decisions

1. **PDF-first editing pipeline**
	The application accepts digital PDFs, extracts structured text regions, and writes approved text back into bounded regions.

2. **Credential modes, not provider modes**
	V1 supports only GitHub Models. The UI exposes:
	- application-managed GitHub Models key
	- user-supplied GitHub Models key for the active request only

3. **SQLite-backed non-production data strategy**
	SQLite is the approved relational store for local development, CI, and hosted QA in V1. Managed PostgreSQL is removed from the V1 QA target to keep the environment cost-bounded. Production database selection is deferred.

4. **QA-only hosted strategy**
	Azure deployment targets one QA environment in `eastus`. Production promotion is deferred, and QA is explicitly optimized for low-concurrency validation rather than scale testing.

5. **Cover letter is an opt-in branch of the same workflow**
	Resume-only remains the default path. Cover letter generation adds one extra generated artifact and one extra review surface.

6. **Human review plus deterministic checks, not LLM-as-judge**
	V1 quality gates rely on schema validation, grounding checks, timing checks, overflow warnings, and human review. LLM-as-judge is optional future work, not a release blocker.

7. **UX spec wins for UI behavior**
	If PRD, ADR, SPEC, and design guidance differ on UI wording or flow, the UX spec is authoritative.

## Options Considered

### Option 1: Multi-provider V1

Pros:
- Broader provider choice

Cons:
- Higher credential complexity
- More failure modes
- More confusing UI and test matrix

### Option 2: GitHub Models-only V1

Pros:
- Smaller surface area
- Cleaner shortlist policy
- Simpler QA and support path

Cons:
- Less provider flexibility

### Option 3: Resume-only V1

Pros:
- Smaller first release

Cons:
- Conflicts with updated product decision

## Rationale

Option 2 plus the cover letter opt-in path is the best round-2 fit because it keeps the user-visible feature set while cutting avoidable provider and environment complexity.

## Consequences

### Positive

- Smaller operational surface
- Cleaner QA strategy
- Removes managed PostgreSQL cost from the V1 QA path
- Less ambiguity for model selection
- Lower coordination cost for implementation

### Negative

- No production deployment path in V1
- No non-GitHub model provider flexibility in V1
- Cover letter adds one more rendering and review surface
- SQLite introduces lower write concurrency and tighter QA scaling constraints

## Implementation Notes

- Default model shortlist is limited to five GitHub Models-hosted models.
- Timeout messaging must branch on credential mode.
- QA infrastructure, smoke checks, and regression checks are part of V1 scope.
- QA deployment must document a writable SQLite database path, serialized migration flow, and lock-contention monitoring expectations before implementation starts.
- Parallel work should be organized around contract boundaries, not shared feature files.

## Governance Gate

- This ADR is accepted for planning alignment only.
- Implementation is blocked until explicit CEO approval is provided.
