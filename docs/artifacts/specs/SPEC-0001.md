## 1. Overview

This specification defines a Django-based resume tailoring application that accepts a PDF resume and job description, generates a tailored resume, optionally generates a cover letter, allows review and editing, and exports PDF output.

User-facing behavior must follow [UX-0001.md](../../ux/UX-0001.md) when wording or interaction order differs from other documents.

## 3. Scope

### In Scope

- Job description input

- Curated top-5 GitHub Models shortlist

### Out of Scope

- Non-PDF editable input
- Non-GitHub model platforms
- LLM-as-judge as a blocking release gate

## 4. User Workflow

1. User uploads a supported digital PDF.
2. User pastes a job description.
3. User chooses credential mode:
	- Use app GitHub Models key
	- Use my GitHub Models key
4. User selects one of the curated GitHub Models.
5. User selects `Resume only` or `Resume and cover letter`.
6. System parses the PDF into ordered sections and bounded regions.
7. System generates tailored content.
8. User reviews and edits all generated content.
9. System exports approved content to PDF.
10. User downloads the output.

## 6. Data Model

### ResumeSession

- Identifier
- Status
- Source PDF path
- Output PDF path
- Output artifact type (`single_pdf` or `dual_pdf`)
- Generation mode (`resume_only` or `resume_and_cover_letter`)
- Credential mode (`app_key` or `user_key`)
- Selected model
- Job description text
- Created timestamp
- Updated timestamp

### ResumeSection

- Session reference
- Section key
- Page number
- Bounding box
- Original content
- Tailored content
- User-edited content
- Review status
- Order index

### CoverLetterDraft

- Session reference
- Original grounding summary
- Tailored content
- User-edited content
- Review status

### GenerationRun

- Session reference
- Model name
- Credential mode
- Prompt version
- Status
- Error code
- Error summary
- Started timestamp
- Completed timestamp
- Duration ms

Note: No raw or encrypted user-supplied key material may be persisted.

Data retention rules:

- Uploaded source PDFs, extracted resume content, job descriptions, edited content, and generated artifacts are encrypted at rest.
- QA sessions are retained for at most 7 days, then purged automatically by a scheduled cleanup job.
- Users may delete the active session and all related artifacts immediately from the review flow.
- Logs, metrics, and traces may include timings and error codes, but must not include resume text, cover letter text, or job-description text.

## 7. Model Strategy

The top-5 shortlist must come from GitHub Models only.

Initial shortlist:

1. GPT-5.1
2. Claude Sonnet 4
3. o3
4. GPT-4.1
5. Llama 4 Maverick

Selection rules:

- Default selection is GPT-5.1.
- The shortlist is configuration-driven.
- Runtime model configuration must pin a concrete GitHub Models identifier and revision date in deployment settings.
- If a model leaves GitHub Models, it must be replaced by another GitHub Models-hosted model before release.

## 8. Timeout and Failure Handling

- If generation times out while using a user-supplied key, show: `Request timed out. Check your API key and retry.`
- If generation times out while using the application key, show: `Request timed out. Retry the request.`
- If the PDF is unsupported or image-only, fail early with a clear rejection message.
- If export cannot fit approved content into a bounded region, block export and flag the section.
- If generation mode is `resume_only`, export one PDF named `tailored-resume.pdf`.
- If generation mode is `resume_and_cover_letter`, expose two separate PDF download actions: `tailored-resume.pdf` and `cover-letter.pdf`.

## 9. Evaluation Strategy

V1 does **not** require LLM-as-judge.

Blocking checks:

- Structured output validation
- Required field completeness
- Known-fact grounding review
- Overflow warning coverage
- p95 generation time <= 120 seconds on supported PDFs
- QA smoke checks pass
- Regression dataset and prompt version are recorded for every blocking run

Non-blocking checks:

- Reviewer-rated usefulness
- Editorial quality feedback

## 10. Deployment Architecture

### Azure Resources

- One Azure Container Apps environment in `eastus`

- One PostgreSQL Flexible Server
- One Redis cache
- One Storage Account
- One Key Vault
- One Container Registry

### Environment Strategy

- Hosted deployment is QA only.
- No production environment is provisioned in V1.
- QA is the validation target for smoke tests and regression checks.

### Deployment Rules

- Use GitHub OIDC where possible.
- Use Key Vault references for operator-managed secrets.
- Keep BYO user key material in memory only for the active request.
- Deploy only to `eastus`.

## 11. Parallel Delivery Plan

### Recommended Engineer Count

1 platform lead for the initial contract freeze, then 5 engineers can work in parallel with low overlap.

### Shared Contracts To Freeze First

1. Session schema
2. Section output schema
3. Cover letter draft schema
4. Generation service interface
5. Export service interface

### Parallel Workstreams

1. **Engineer A - Upload and session lane**
	Owns form submission, session creation, file validation, storage wiring.

2. **Engineer B - PDF ingestion lane**
	Owns PDF parsing, section detection, bounding boxes, supported-file checks.

3. **Engineer C - Generation lane**
	Owns GitHub Models client, shortlist config, prompt execution, timeout handling.

4. **Engineer D - Review and export lane**
	Owns review UI bindings, edit persistence, cover letter review surface, export flow.

5. **Engineer E - QA lane**
	Owns Bicep, GitHub Actions, smoke checks, evaluation runner, deployment validation.

### Lowest-Conflict Sequencing

1. Freeze schemas and interfaces.
2. Land session and ingestion foundations.
3. Start generation, review/export, and QA lanes in parallel.
4. Merge through contract tests, not ad hoc shared-file edits.

## 12. Acceptance Criteria

- User can generate resume-only output.
- User can generate resume-and-cover-letter output.
- Cover letter is absent when not selected.
- No ZIP package is produced in any generation mode.
- Only GitHub Models-hosted shortlist models appear.
- QA deployment runs in `eastus` only.
- Timeout messaging matches credential mode.
- Human review is mandatory before export.
