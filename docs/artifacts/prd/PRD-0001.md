# PRD: ResumeTailor V1

**Epic**: #0001
**Status**: Approved (Planning Only)
**Author**: GitHub Copilot
**Date**: 2026-04-01

## 1. Product Summary

ResumeTailor helps a user upload a PDF resume, paste a target job description, generate a tailored resume, optionally generate a cover letter, review and edit the result, and export approved PDFs. V1 is single-user and local-first, with one hosted QA deployment in Azure.

For user-facing workflow, field labels, interaction order, and error messaging, [UX-0001.md](../../ux/UX-0001.md) is the source of truth.

## 2. Problem Statement

Job seekers spend too much time rewriting resumes and cover letters for each role. Manual tailoring is slow, inconsistent, and often damages formatting.

## 3. Goals

1. Produce a tailored resume draft in under 2 minutes for supported PDFs.
2. Let the user choose `Resume only` or `Resume and cover letter` before generation.
3. Preserve page size and untouched visual elements for supported PDFs.
4. Keep the user in control through a mandatory review and edit step before export.
5. Limit hosted infrastructure to one QA environment in `eastus` for V1.

## 4. Non-Goals

- Production deployment in V1
- Multi-user accounts or tenancy in V1
- Non-PDF editable input in V1
- Provider-specific model catalogs outside GitHub Models in V1

## 5. Target Users

- Individual job seekers tailoring their own resume
- Career coaches preparing reviewed drafts with a client present

## 6. Functional Requirements

### P0

1. Accept digital PDF resume upload.
2. Accept pasted job description text.
3. Expose a generation mode selector with two choices: `Resume only` and `Resume and cover letter`.
4. Generate tailored resume section content grounded only in the uploaded resume and job description.
5. Generate a cover letter only when the user selects the cover letter option.
6. Show original and tailored content for review and allow edits before export.
7. Export approved output after review: a single resume PDF for `Resume only`, or two separate PDF download actions (`tailored-resume.pdf` and `cover-letter.pdf`) for `Resume and cover letter`.
8. Use GitHub Models only for model access in V1.
9. Offer two credential modes:
	 - Application-managed GitHub Models key
	 - User-supplied GitHub Models key, used only for the active request and never persisted
10. If generation times out while using a user-supplied key, the UI must tell the user to check their API key and retry.

### P1

1. Show a change summary between original and tailored content.
2. Warn before export if a section risks overflow.
3. Record generation timing and failure metadata without logging sensitive content.

## 7. Model Policy

V1 uses a curated top-5 model shortlist from GitHub Models only. Initial shortlist:

1. GPT-5.1
2. Claude Sonnet 4
3. o3
4. GPT-4.1
5. Llama 4 Maverick

The shortlist is configuration-driven and must stay within the GitHub Models catalog.

## 8. Success Metrics

| Metric | Target |
|--------|--------|
| End-to-end generation time | p95 <= 120 seconds on supported PDFs |
| Factual invention | 0 tolerated in blocking review set |
| Structured output validity | 100 percent for blocking fields |
| Overflow warning coverage | 100 percent of known overflow cases flagged |
| QA deployment health | All smoke checks pass |

## 9. Constraints

- Django is the application framework.
- PDF is the only editable source format in V1.
- GitHub Models is the only model platform in V1.
- Hosted deployment is QA only.
- Azure region is `eastus` only.
- The app remains single-user and local-first for V1 runtime assumptions.

## 10. Risks and Mitigations

- **Risk**: PDF fidelity varies by source file.
	**Mitigation**: Use bounded region replacement, overflow warnings, and supported-PDF gating.
- **Risk**: AI invents facts.
	**Mitigation**: Structured prompts, deterministic validation, and mandatory human review.
- **Risk**: User-supplied key failures are misdiagnosed.
	**Mitigation**: On timeout, explicitly ask the user to check their API key and retry.
- **Risk**: Scope expands into production hardening too early.
	**Mitigation**: Lock V1 to one QA environment in `eastus`.

## 11. Council Recommendations Adopted In Round 2

1. Close product boundary decisions explicitly.
2. Make PDF fidelity and latency measurable.
3. Replace vague provider logic with GitHub Models-only scope.
4. Add operational deployment detail for QA.
5. Add smaller delivery slices so engineers can work in parallel.
6. Treat the UX design spec as the source of truth for user-facing behavior.

## 12. Parallel Delivery Guidance

After a short foundation pass, up to 5 engineers can work in parallel with low overlap:

1. Session and upload lane
2. PDF ingestion and section-mapping lane
3. Model orchestration and generation lane
4. Review, edit, and export lane
5. QA infrastructure and validation lane

Shared contracts that must be frozen first:

- Resume session schema
- Section output schema
- Generation request and response contract
- Export artifact contract
